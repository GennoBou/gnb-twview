# gnb-twview v0.1.0 開発計画・技術調査レポート

本ドキュメントは、Twitch専用ブラウザ `gnb-twview` (v0.1.0) の開発に向けた技術的調査、アルゴリズムの検討、および実装への具体的な提案をまとめたものです。

---

## 1. Twitch 埋め込みプレイヤーと `parent` パラメータの仕様

Twitchの埋め込みプレイヤー（`player.twitch.tv`）は、セキュリティ（クリックジャッキング対策など）のため、埋め込み先の親ドメインを `parent` パラメータとして明示することを要求します。

### Wails v3 における挙動と対策
- **Windows (WebView2)**: Wails v3 のアセットサーバーは、デフォルトで `localhost` または `wails.localhost` でフロントエンドをホストします。
  - 例: `https://player.twitch.tv/?channel=<username>&parent=localhost`
  - Wails側のホスト設定でホスト名を `localhost` に固定することで、このパラメータは `localhost` のままで安定して動作します。
- **Android (WebView)**: Android でローカルアセットを直接 `file://` スキームで読み込むと、Originが `null` になるためTwitchプレイヤーの再生が拒否されます。
  - **対策**: Android の WebView で `WebViewAssetLoader`（またはWails v3のAndroidアセットサービング）を使用し、フロントエンドを仮想ドメイン（例: `localhost` や `appassets.androidplatform.net`）上でホストさせます。そのホスト名を `parent` に指定することでエラーを回避できます。

## 2. ミュート機能（実装の見送り）

当初はブラウザのアウトプット音量を直接制御する独自のミュートボタンの実装を計画していましたが、この機能を廃止し、音量管理はTwitch側の標準機能に委ねる決定をしました。

### 経緯と理由
動画再生エリアが公式の `twitch.tv` ページを直接表示する構成に変更されたため、ミュート状態はTwitch自身の `localStorage`（キー: `video-muted`）によってネイティブに保存・復元されます。したがって、アプリケーション側に独自のミュートボタンや同期ロジックを持つ必要性がなくなり、Twitchプレイヤー本体のボリュームおよびミュート制御に一本化しました。

## 3. Twitch API と OAuth ログイン・認証管理

ユーザー権限でのAPI利用およびClient IDの管理方法について。

### 認証フローの設計 (OAuth 2.0 PKCE)
- **Client ID と Secret の扱い**:
  - `Client ID` は一般に公開されても問題ありませんが、`Client Secret` は絶対にバイナリや公開リポジトリに含めてはいけません。
  - よって、Secretを必要とせず安全に認証が可能な **OAuth 2.0 Authorization Code Flow with PKCE** を採用します（当初検討していた Implicit Grant はセキュリティ非推奨のため除外されました）。
- **ログイン手順**:
  1. アプリ側で一時的な `code_verifier` と `code_challenge` を生成する。
  2. アプリ内で Twitch の認証 URL（`https://id.twitch.tv/oauth2/authorize`）に `code_challenge` などのパラメータを付与し、外部ブラウザで開く。
  3. Wails アプリ内で一時的にローカルHTTPサーバー（例: `http://localhost:8520/callback`）を起動し、認可コード（`code`）を受け取る。
  4. Goバックエンドから直接 Twitch サーバーへ POST リクエストを送り、認可コードと `code_verifier` を引き換えてアクセストークンを取得する。
  5. トークン取得後、ローカルサーバーを停止し、トークンを安全な領域に保存する。
- **トークンの保存方法**:
  - 現在はホームディレクトリ配下の `~/.gnb-twview/token.json` に保存しています。将来的に、Windows: `Windows Credential Manager`（資格情報マネージャー）や Android: `EncryptedSharedPreferences` を使用した暗号化・セキュア保存への移行を検討します。

## 4. オートモードの巡回アルゴリズム提案

ユーザーが悩まれている「キューの割り込みで初回値未満の配信者が固まる問題」を解決し、より快適な視聴体験を提供するアルゴリズムを提案します。

### スマート・ラウンドロビン (Smart Round-Robin) アルゴリズム
視聴履歴を「累積視聴秒数」として管理し、以下のルールでキューを構築・ソートします。

1. **状態管理**:
   - `TargetTime`（目標視聴時間。6分固定の定数扱い）
   - 配信者ごとの `TotalWatchedTime`（累積視聴秒数。アプリ再起動時に最終視聴日時が3時間以内のデータは自動で引き継ぎ、3時間を超えた場合は破棄）
   - 最終視聴日時 `LastWatchedAt` (配信者ごとに個別に追跡管理し、3時間経過でメモリおよびファイルから自動クレンジング)
2. **優先度ソート（巡回順の決定）**:
   - ライブ中の配信者リストを取得するたび、以下の優先順位でソートしてキューを構成します。
     - **第1優先**: `TotalWatchedTime < TargetTime` の配信者（目標未達グループ）
       - その中では、`TotalWatchedTime` が**少ない順**に並べる（未視聴の配信者を最優先）。
     - **第2優先**: `TotalWatchedTime >= TargetTime` の配信者（目標達成グループ）
       - その中では、`TotalWatchedTime` が少ない順、またはランダムで並べる。
3. **滞在時間（巡回タイマー）の動的切り替え**:
   - 現在表示している配信者が：
     - **目標未達**（`TotalWatchedTime < TargetTime`）の場合：**「目標時間（6分） － 累積視聴秒数」**で算出される残りの時間だけ滞在する。
     - **目標達成**（`TotalWatchedTime >= TargetTime`）の場合：**「巡回時間（初期値3分、設定画面から変更・保存可能）」**滞在する。
4. **効果**:
   - まだ見ていない配信者がキューの前方に並び、かつ「最も見ていない人」から順に視聴されるため偏りが生じません。
   - スキップや再起動を行っても、すでに視聴した時間が引き継がれ、残りの秒数から滞在カウントダウンが再開されるため、巡回効率が劇的に向上します。
   - 一通り全員を6分見終わった後は、自動的に設定された巡回時間（デフォルト3分）ごとのスピーディーな巡回モードに移行します。

## 5. プレイヤー表示方法（CSS除外 vs player.twitch.tv）

### 比較と最終デザイン決定
- **選択肢1: `player.twitch.tv` (埋め込みプレイヤー) [不採用]**
  - **デメリット**: 埋め込みプレイヤーでの視聴では、視聴ストリークやチャンネルポイントが一切獲得できない仕様上の問題があります。また、埋め込みフレーム内からのログインがTwitch側の制限で難しく、セッション共有処理が複雑化します。
- **Option 2 (選択): カスタムCSSインジェクションを伴う `twitch.tv/<channel>` の直接ロード (TwitchWindow)**
  - **メリット**: 公式サイトをそのまま読み込むため、視聴ストリークやチャンネルポイントが正常に獲得できます。公式のヘッダーから簡単にTwitchにログインでき、ログインセッションも共有されます。
  - **デメリットへの対策 (要素の非表示と位置固定)**: CSSを `/assets/inject.css` に切り出して動的ロードさせ、かつSPA/Reactの再描画対策として `MutationObserver` と `setInterval` を組み合わせた自己修復型CSSインジェクションスクリプトを実装しました。また、WailsのIPC通信境界をバイパスする readyシグナルの偽装ハック (`HandleMessage("wails:runtime:ready")`) を適用しています。
  - **マルチウィンドウ設計（設定ウィンドウの分離）**: 設定開閉時にZオーダーの操作や非表示/画面外退避を行うと、OSおよびWebView2の仕様上、配信画面（`TwitchWindow`）のリサイズや再描画が発生してもたつきが生じます。これを解決するため、設定モーダルを独立した「設定専用ウィンドウ」としてポップアップさせます。同じフロントエンドアセットをクエリパラメータ（`/?mode=settings`）付きで読み込み、常に最前面に表示させることで、設定操作中も配信画面へのOS操作を一切行わず、滑らかな再生と操作を維持します。

## 6. クロスプラットフォーム開発（Wails v3 と Android）

Wails v3 は現在 Android サポートを進めていますが、デスクトップ（Windows）とモバイル（Android）で完全に同じGoコードを共有してビルドするには、環境構築やAPI差異の面でハードルがあります。

### 推奨する開発アプローチ
1. **ロジックのフロントエンド集約**:
   - キュー管理、タイマー、Twitch API通信、OAuthフローなどのコアロジックを、Go側ではなく **Svelte (TypeScript) 側** で実装します。
   - Go側は「Windowsのネイティブ窓操作」などのOS固有機能のみを担当させます。
2. **Androidへの移植戦略**:
   - v0.1.0（Windows版）のリリース後、Svelteのコードベースをほぼそのまま用いて、Android版は **Capacitor** または **Tauri v2 (モバイル対応が安定)** を用いてパッケージングします。
   - これにより、プラットフォーム固有のWebViewの差異に悩まされることなく、短期間で高品質なAndroid版を開発できます。
